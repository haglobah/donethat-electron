# Release Integrity

This document explains how DoneThat Desktop maps published binaries back to source and what integrity checks exist in the release flow.

## Source-To-Binary Mapping

- The canonical source repository is `donethatai/donethat-electron`.
- Published binaries and update assets are uploaded to `donethatai/donethat-releases`.
- Release uploads run `npm run release:prepare` before `electron-builder --publish always`.
- `release:prepare` runs `scripts/ensure-release-tag.js`, which:
  - reads the version from `package.json`
  - resolves the expected tag name as `v<version>`
  - verifies that the tag points at `HEAD`, or creates it if missing
  - pushes the tag to `origin` before the upload continues

This means the release version and the source tag are locked together by the release automation.

## Release Workflow Guardrails

- The GitHub Actions release workflow runs only from the `main` branch.
- The workflow checks out the full git history so tags can be validated and pushed.
- GitHub Actions configures a dedicated bot identity for release tagging.
- Platform upload commands all run through the same tag-guarded release flow.

## Signing And Verification

### macOS

- macOS release builds use Apple signing credentials provided through GitHub Actions secrets.
- Electron Builder runs the `afterSign` hook in `scripts/notarize.js`.
- The notarization step uses App Store Connect API credentials when available, with Apple ID credentials as a fallback.

### Windows

- Windows release builds use Azure Trusted Signing (Artifact Signing).
- The workflow authenticates to Azure via OIDC federated identity (`azure/login@v2`) and provisions the Trusted Signing dlib plus a `metadata.json` pointing at the `DoneThat / Letss` certificate profile in West Europe.
- `scripts/azure-sign-windows.js` signs the produced artifact with `signtool sign /dlib /dmdf` against `http://timestamp.acs.microsoft.com` and immediately verifies it with `signtool verify /pa /v`.
- Windows arm64 native code is compiled on the `windows-11-arm` runner as an unpacked app. The unpacked app is then signed and packaged on a `windows-latest` x64 runner because Azure Trusted Signing currently ships an x64 dlib, not an ARM64-native dlib.
- Windows arm64 installer and updater metadata are generated after the unpacked app has been signed, so published hashes describe the final signed artifacts.

### Linux

- Linux AppImage builds are produced by the same release workflow.
- No separate Linux signing flow is configured in this repository today.

## Checksums And Update Metadata

- Electron Builder is configured with `generateUpdatesFilesForAllChannels: true`.
- Published updater metadata generated by Electron Builder includes SHA-512 hashes for updater assets.
- Those update files are part of the release output and should be published alongside the binary artifacts in `donethatai/donethat-releases`.
- This repository documents integrity hashes through Electron Builder updater metadata rather than through separate standalone checksum files.

## How To Verify A Release

1. Confirm the app version in the published binary matches the git tag `v<version>` in `donethatai/donethat-electron`.
2. Confirm that tag points at the source commit intended for release.
3. Review the corresponding release assets in `donethatai/donethat-releases`.
4. Verify platform signatures where applicable:
   - macOS: use standard `codesign` and `spctl` checks
   - Windows: use `signtool verify /pa /v` or Windows Authenticode inspection tools
5. Compare updater metadata hashes against the published updater assets when validating auto-update artifacts.

## Reproducibility Notes

- The release flow is documented and constrained, but this repository does not claim bit-for-bit reproducible builds.
- For the closest rebuild match, use:
  - Node.js `22`
  - the same platform target as the published artifact
  - the same `build:*` or `upload:*` script family from `package.json`
  - the same signing/notarization credentials where applicable
